# 《夜巡录：荒庙篇》实现细节与协作复盘

## 1. 当前版本事实

《夜巡录：荒庙篇》是一个运行在浏览器里的单人卡牌构筑 roguelike demo。当前版本不需要后端服务，构建后就是一组静态文件。

技术栈：

- React 18：承载页面状态、复杂 UI、卡牌、地图、商店、事件、结算等界面。
- Phaser 3.90：承载战斗舞台、背景、人物和怪物站位、受击摇晃、屏幕震动、雾气粒子。
- TypeScript：定义游戏状态、卡牌、敌人、遗物、地图节点等数据结构。
- Vite 8：开发服务器与静态构建。
- lucide-react：少量通用 UI 图标，例如声音、回首页、重新开始等。

主要目录：

```text
src/
  App.tsx                 React UI、屏幕切换、拖拽出牌、素材引用
  styles.css              全局视觉、卡牌、HUD、地图、商店、事件、响应式布局
  game/
    types.ts              游戏状态与内容类型
    content.ts            卡牌、敌人、遗物、事件、节点定义
    engine.ts             纯逻辑状态机
    audio.ts              背景音乐与音效控制
  phaser/
    CombatStage.tsx       React 包装的 Phaser 战斗舞台

assets/
  audio/                  BGM 与音效
  generated/              生成或制作后的正式战斗图、怪物图、结算视频
  vendor/                 从素材包里挑选并整理后的 UI 素材

docs/
  PLANNING.md             早期玩法规划
  FIRST_ACT_DEMO_ROADMAP.md
  IMPLEMENTATION_DETAILS.md
  PACKAGING_DISTRIBUTION.md
```

## 2. 架构选择

最初可以做成一个纯 HTML/CSS/JS 的小原型，但后面 UI 复杂度很快上来了：卡牌扇形手牌、拖拽出牌、地图分岔、商店、事件、结算视频、难度选择、顶部 HUD、底部牌堆，这些都更适合交给 React。

战斗舞台则不同。角色、怪物、背景、粒子、受击晃动和镜头震动如果全部用 DOM 做，会让样式和状态变得很拧巴。Phaser 更适合做这部分，所以最终选择了 React + Phaser 混合：

- React 管 UI，保证复杂界面可维护。
- Phaser 管战斗舞台，保证画面反馈和动画更自然。
- 游戏逻辑放在 `src/game/engine.ts`，避免 UI 和规则互相缠死。

这个选择的关键不是“技术更炫”，而是职责边界清楚：卡牌游戏的操作层很复杂，舞台表现也很复杂，把它们硬塞进同一套抽象里反而会拖慢迭代。

## 3. 状态模型

核心状态定义在 `GameState`：

- `screen`：当前屏幕，例如 title、map、combat、cinematic、reward、shop、event、rest。
- `player`：生命、能量、金币、牌组、遗物、状态。
- `mapNodes`、`availableNodeIds`、`visitedNodeIds`：路线地图与可选节点。
- `combat`：当前战斗，包括抽牌堆、弃牌堆、消耗堆、手牌、敌人、回合数、受击目标。
- `cinematic`：战斗胜利后的结算过场状态。
- `reward`、`event`、`shop`：非战斗模块的临时状态。
- `seed`：伪随机种子。
- `lastFx`：给 UI 层触发音效的轻量信号。

React 里通过 `transact(fn)` 修改状态。它会先 `structuredClone` 出一个 draft，再调用 engine 里的纯逻辑函数，最后一次性 `setGame`。这样做有几个好处：

- 逻辑函数可以像传统游戏状态机一样直接改 draft。
- React 仍然拿到新的对象引用，可以正常触发渲染。
- 音效通过 `lastFx` 从规则层传给 UI 层，不需要 engine 直接操作浏览器 Audio。

## 4. 游戏规则实现

规则集中在 `src/game/engine.ts`。当前包括：

- 开局：根据难度生成血量、金币、能量、初始牌组、起始遗物。
- 地图：生成 8 行路线节点，普通战、精英、事件、商店、休整、Boss 分布在不同层。
- 战斗：抽牌、洗弃牌堆、出牌、结算敌人意图、符印结算、胜负判断。
- 奖励：金币、卡牌三选一、精英/Boss 遗物。
- 事件：不同选择影响生命、金币、牌组、遗物。
- 商店：买牌、买遗物、删牌。
- 休整：回血或升级。

卡牌效果目前是手写分支，而不是独立 DSL。这是一个刻意选择：demo 阶段卡牌数量有限，手写分支更直接，调数值和查 bug 更快。等内容扩到几十上百张牌，再把效果拆成数据驱动 DSL 会更划算。

## 5. 卡组循环与数值思路

这个 demo 的核心爽感不是单张牌很大，而是让牌组循环起来。

当前循环支撑来自几类牌：

- 0 费抽牌：`清心咒`、`回风符`。
- 抽防一体：`云步`、`驱雾`。
- 弃牌堆回卷：`灰烬回卷`。
- 能量补偿：`丹砂炼气`、`破邪`。
- 长线能力：`夜眼`、`请城隍`。
- 爆发出口：`焚符`、`引雷`、`五雷正法`。

调数值时主要考虑三件事：

1. 第一关不能逼玩家必须拿到某一张牌才能过。
2. 玩家应该能看懂自己为什么变强，例如“符印越叠越多，然后焚符一口气爆掉”。
3. 演示难度要允许录视频时顺利打通，所以给了更高血量、更多初始资源和循环遗物。

这也是后来加入难度选择的原因。标准难度保留完整体验，演示难度服务录屏和传播，劫难难度用于压测数值。

## 6. 拖拽出牌

出牌方式从点击改成拖拽，是为了贴近《杀戮尖塔》的身体感：玩家把牌从手里“推出去”，而不是点一个按钮。

实现上，`GameCard` 接收 pointer 事件：

- `pointerdown`：记录牌 uid 和起始坐标。
- `pointermove`：计算拖拽偏移，卡牌跟随鼠标。
- `pointerup`：根据释放坐标判断是否命中玩家或敌人目标区。

目标判断由卡牌类型决定：

- 攻击牌需要拖到敌人区域。
- 技能/法门牌需要拖到玩家区域。
- 不可打出或能量不足的牌直接禁用。

拖拽中 React 会显示：

- 中央施放提示。
- 玩家侧“加护预备”反馈。
- 敌人侧“受击预热”反馈。
- 释放成功后有 shield 或 strike burst。

这部分的关键不是复杂算法，而是把“玩家正在瞄哪里”清楚地反馈出来。卡牌游戏的交互如果缺少这个反馈，会显得像网页表单。

## 7. React 与 Phaser 的同步

Phaser 舞台没有直接持有完整 `GameState`。React 每次渲染时把战斗信息压成一个 `StageSnapshot` 传进去：

- 玩家血量、格挡。
- 敌人 id、名称、血量、格挡、符印、意图。
- `pulse` 和 `hitTarget`。

Phaser scene 收到 snapshot 后更新贴图、尺寸、位置和文本。如果 `pulse` 变化，就触发镜头震动和对应角色左右摇晃。

这样做可以避免两个状态源：

- React 是唯一的游戏状态来源。
- Phaser 只是视觉投影。
- 如果以后 Phaser 出问题，规则层不受影响。

## 8. 视觉资产管线

项目里有三类素材来源：

1. 现成素材包：原始文件放在 `素材/`，不进入 Git；筛选后进入 `assets/vendor/`。
2. 生成资产：敌人大立绘、主角、背景、结算海报和视频放在 `assets/generated/`。
3. 代码样式：HUD、布局、暗场、卡牌排布、拖拽反馈由 CSS 和 React 组合完成。

素材处理原则：

- 原始大素材包不入库，避免 Git 爆炸。
- 只把游戏实际用到的素材放进 `assets/`。
- 卡牌边框、费用宝石、牌堆、血量、遗物等高频 UI 优先替换成素材图。
- 中心画面尽量少用半透明描边和网页感底色，避免破坏沉浸。

最近一次费用 UI 就是这个原则的例子：早期费用是代码画的方框，后来换成 `法宝奇遇图标` 里的宝石，并按费用高低分档。

## 9. 结算视频管线

游戏内已经支持击败敌人后进入 `cinematic` 屏幕。

流程：

1. 敌人 HP 降到 0。
2. `engine.ts` 创建 `CinematicState`。
3. React 进入 `CinematicScreen`。
4. 如果有对应视频，播放视频。
5. 视频结束或玩家点击跳过，进入奖励或胜利页。
6. 如果视频缺失，使用静态结算画面 fallback。

这套设计服务 Seedance 2.0 的工作流：先用当前角色和敌人的参考图生成短视频，再把压缩后的视频放进 `assets/generated/cinematics/`。普通怪视频可以短，Boss 视频可以长一些。

## 10. 声音实现

音频集中在 `src/game/audio.ts`：

- 标题页 BGM 音量更大。
- 进入游戏后 BGM 降低，作为背景氛围。
- 点击、攻击、格挡、火焰、雷击、气功等反馈按 `lastFx` 触发。
- 视频过场期间可以继续扩展为 ducking，也就是自动压低背景音乐。

音效的设计目标是“操作有重量”。即使画面还在迭代，只要卡牌、攻击、受击、奖励有声音，游戏就会立刻更像一个完整作品。

## 11. 分发前修正

为了适配客户端打包和本地文件加载，当前做了几个发布向调整：

- `vite.config.ts` 设置 `base: "./"`，保证 Electron 用 `file://` 加载 `dist/index.html` 时资源路径不失效。
- CSS 里的背景图从 `/assets/...` 改为相对路径，让 Vite 能正确打包并重写资源 URL。
- 新增 `electron/main.cjs`，用 Electron 包一层桌面窗口。
- 新增 `desktop:pack` 和 `desktop:dist` 脚本，分别用于本地验证和正式出包。
- GitHub 的角色调整为 Releases 分发安装包，而不是只做网页试玩。

同时加入了 `public/favicon.png`、`desktop-assets/icon.png` 和 `desktop-assets/icon.ico`。当前使用游戏内金色八卦图标作为临时图标，正式发布前建议再生成一枚专用图标。

## 12. AI 与非游戏开发者如何协作完成

这个项目不是一次性“写完”的，而是按可玩性一路迭代出来的。

实际过程大致是：

1. 先确定玩法母题：不是直接照搬《杀戮尖塔》，而是把题材换成志怪、荒庙、城隍印、符箓、香火。
2. 先做能跑通的原型：标题、地图、战斗、奖励、Boss，哪怕 UI 粗糙，也先让完整循环存在。
3. 再修核心体验：牌组循环、分岔路线、拖拽出牌、难度选择。
4. 再做视听质感：主角/怪物立绘、背景视频、结算视频、按钮素材、费用宝石、音效。
5. 再做细节打磨：血条遮挡、按钮重影、HUD 文字统一、顶部背景漏月亮、拖拽反馈。
6. 最后转向分发：Electron 客户端、GitHub Releases、README、文档、图标、宣传截图。

用户不需要懂代码，但需要持续指出“哪里不像游戏”。AI 的工作是把这些感觉翻译成工程任务：

- “这里糙”可能对应布局层级、素材分辨率、边框样式或透明背景。
- “拖出去没有感觉”对应 pointer 事件、目标热区、hover feedback、释放成功特效。
- “卡组循环不起来”对应抽牌、能量、弃牌堆回卷、奖励池权重和敌人数值。
- “看着尴尬”可能是背景图层重叠、亮度冲突或视觉焦点错位。

这种协作方式很适合小型游戏 demo：人负责审美判断、体验判断和方向感，AI 负责快速试错、改代码、查资源、跑构建、验证浏览器画面。

## 13. 目前技术债

当前 demo 已经可玩，但还有一些明确技术债：

- 卡牌效果仍是手写分支，内容扩张后需要 DSL 或表格驱动。
- 商店、事件、休整仍可继续组件化。
- 视频资源体积较大，后续应压缩、懒加载或拆包。
- 没有存档、每日种子和设置持久化。
- 没有自动化测试；目前主要靠 `npm run build` 和浏览器手测。
- 素材授权需要在公开发布前逐项确认。

这些技术债不阻塞 demo 分发，但会影响继续扩展成更大项目。
